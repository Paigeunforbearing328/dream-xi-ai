# 安全政策 — Fair Play 守则（Security Policy）

感谢你帮助维护 Dream XI AI 球场的安全。

---

## 支持的版本

我们仅对最新主分支（`main`）的最新发布版本提供安全修复支持。

| 版本 | 支持状态 |
|------|----------|
| 最新 `main` 分支 | ✅ 持续支持 |
| 旧版本 | ❌ 不再支持 |

如果你使用的是旧版本并发现漏洞，请先升级到最新版后确认问题是否仍然存在。

---

## 报告安全漏洞

**请勿通过公开的 GitHub Issues、Discussions 或 Pull Requests 报告安全漏洞。**

这不是场上公开犯规——安全漏洞需要私密处理。

如果你发现任何安全漏洞，请通过以下方式进行负责任披露：

- 通过 [GitHub Security Advisories](https://github.com/loulanyue/dream-xi-ai/security/advisories/new) 私密报告

### 报告时请尽量包含以下信息

- 漏洞类型（如：路径遍历、不当权限提升、依赖注入等）
- 漏洞所在的源文件路径（含行号或函数名）
- 受影响的版本（tag、branch 或 commit SHA）
- 触发漏洞所需的特殊配置（如有）
- 最小化复现步骤（step-by-step）
- 概念验证代码或利用示例（如可提供）
- 漏洞影响评估

以上信息将帮助我们更快地进行"VAR 回放"（漏洞分类处理）。

---

## 处理流程与时效

| 阶段 | 时效 | 说明 |
|------|------|------|
| 确认收到 | 3 个工作日 | 发送确认回复 |
| 初步评估 | 7 个工作日 | 完成影响评估并告知结论 |
| 修复与发布 | 视严重程度 | 从数小时到数周不等 |
| 公开披露 | 修复后 | 在 CHANGELOG.md 中注明安全修复条目 |

## 球队铁律与安全

Dream XI 的四条铁律（Fair Play Rules）本身就是安全的第一道防线：

1. **数据圣殿** — 不删除持久化数据
2. **进程自保** — 不杀死父进程
3. **配置只读** — 不修改运行时配置
4. **端口边界** — 不跨越服务边界

这些规则在提示层和代码层双重执行，确保即使 AI Agent 被对抗性输入攻击，也不会造成不可逆损害。

## 依赖安全

Dream XI 通过以下方式持续监控依赖安全：

- GitHub Dependabot 自动依赖更新提醒
- CI 流水线中集成安全扫描
- 定期审查第三方依赖

如果你在依赖项中发现漏洞，也请通过上述渠道告知我们。
