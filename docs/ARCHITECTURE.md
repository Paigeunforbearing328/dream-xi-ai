# 架构决策记录（Architecture Decision Records）

本文档记录 Dream XI AI 的关键架构决策，采用 ADR（Architecture Decision Record）格式。每条记录说明"决定了什么"、"为什么这样决定"以及"放弃了什么替代方案"。

---

## ADR-001：三层架构原则

**状态：** ✅ 已采用  
**日期：** 2026-06-18

### 决定

Dream XI 采用严格的三层架构：**模型层 → Agent CLI 层 → 平台层**，每层职责不重叠。

### 背景

早期设计中，平台层试图承担部分推理任务（如意图识别、任务分解）。这导致平台与模型能力强绑定，每次换模型都需要重写平台逻辑。

### 决定理由

| 层 | 负责 | 不负责 |
|----|------|--------|
| **模型层** | 推理、生成、理解 | 长期记忆、协作纪律 |
| **Agent CLI 层** | 工具使用、文件操作、命令执行 | 团队协调、跨模型审查 |
| **平台层（Dream XI）** | 身份、协作、纪律、审计 | 推理（那是模型的工作） |

> *Models set the skill ceiling. The platform sets the tactical floor.*

### 放弃的替代方案

- **单层 Orchestrator**：把所有逻辑塞进一个 Boss Agent。问题：单点故障，无法跨模型协作。
- **平台内置推理**：平台自己做意图识别。问题：和模型能力重复，且不如模型做得好。

## ADR-002：持久身份（Persistent Identity）策略

**状态：** ✅ 已采用  
**日期：** 2026-06-18

### 决定

每个 Agent 在跨会话、跨上下文压缩时保持稳定的身份（角色、个性、记忆锚点）。

### 背景

LLM 的上下文窗口是有限的。当对话过长，早期的角色设定和记忆会被"压缩掉"，导致 Agent 失去一致性——一个本应"严谨审慎"的审查 Agent 突然开始随意附和。

### 解决方案

1. **身份锚定**：每次请求头部注入精简的角色卡（< 200 tokens），包含名字、编号、核心职责、性格关键词。
2. **记忆分层**：
   - **工作记忆**（当前线程）：完整上下文
   - **情景记忆**（跨线程）：压缩的决策摘要，存 Redis
   - **语义记忆**（持久）：经验沉淀，以 Markdown 文档形式存储
3. **压缩感知**：当检测到上下文接近窗口上限时，触发"赛后总结"自动压缩并归档。

### 放弃的替代方案

- **无状态设计**：每次对话从零开始。问题：无法积累团队经验，协作一致性差。
- **完整对话存储**：保存所有对话记录并每次注入。问题：Token 消耗爆炸，成本不可控。

## ADR-003：A2A 消息路由设计

**状态：** ✅ 已采用  
**日期：** 2026-06-18

### 决定

采用 `@mention` 显式路由 + 自动意图路由双模式，消息通过统一消息层分发。

### 路由优先级

```
显式 @mention > 任务类型意图识别 > 默认路由（Leo #10）
```

### 线程隔离

每个 Thread 是独立的上下文空间。Agent 只能看到当前 Thread 的消息，不会跨 Thread 泄露上下文。

### 结构化交接协议

当 Agent 将任务传给另一个 Agent 时，必须附带标准交接摘要（见 `docs/SOP.md`），避免上下文丢失。

### 放弃的替代方案

- **广播模式**：所有消息发给所有 Agent。问题：Token 浪费，每个 Agent 都在处理无关信息。
- **纯自动路由**：完全由系统决定发给谁。问题：用户无法精准控制，透明度差。

## ADR-004：MCP 集成策略

**状态：** ✅ 已采用  
**日期：** 2026-06-18

### 决定

通过 **MCP Callback Bridge** 实现跨 Agent 工具共享，支持非 Claude 模型通过回调机制使用 MCP 工具。

### 背景

MCP（Model Context Protocol）是 Claude 原生支持的工具协议。GPT、Gemini 等模型原生不支持 MCP，但 Dream XI 需要所有球员共用工具（如文件系统、代码执行、数据库查询）。

### 解决方案

```
Claude (MCP 原生) ──────────────────┐
GPT/Codex (非 MCP) → Callback Bridge → MCP Tool Server
Gemini (非 MCP)    ──────────────────┘
```

Callback Bridge 将非 MCP 模型的工具调用转换为标准 MCP 请求，统一工具访问层。

### 放弃的替代方案

- **各模型独立工具层**：每个模型用自己的工具调用格式。问题：无法共享工具，代码重复。
- **仅支持 Claude**：只用 Claude MCP。问题：锁定单一模型，失去多模型协作的价值。

## ADR-005：球队铁律（Fair Play Rules）实现

**状态：** ✅ 已采用  
**日期：** 2026-06-18

### 决定

在**提示层**和**代码层**双重实现四条铁律，确保 AI Agent 行为安全边界。

### 四条铁律

1. **数据圣殿** — 不删除/清空持久化存储
2. **进程自保** — 不杀死父进程或修改启动配置
3. **配置只读** — 不修改运行时配置文件
4. **端口边界** — 不访问不属于自己服务的端口

### 双重执行机制

```
提示层：系统 Prompt 中明确声明铁律 + 每次对话注入提醒
代码层：危险操作前的权限检查 + 拦截器中间件
```

### 设计哲学

铁律不是"禁止令"，而是球队自愿遵守的 **Fair Play 约定**。这一定性影响了 Agent 的态度：
- ❌ 错误定性：「系统不允许我删除数据库」（被动限制）
- ✅ 正确定性：「我们不删自己的数据库——那是比赛记录」（主动认同）

## ADR-006：Redis 可选化（`--memory` 模式）

**状态：** ✅ 已采用  
**日期：** 2026-06-18

### 决定

Redis 作为**可选依赖**，通过 `--memory` 标志启用纯内存模式，降低首次安装门槛。

### 权衡

| 模式 | 优点 | 缺点 |
|------|------|------|
| Redis 模式 | 跨重启持久化，生产可用 | 需要额外安装 Redis |
| 内存模式 | 零依赖，5 分钟上手 | 重启后记忆丢失 |

### 目标用户

- **内存模式**：本地探索、功能演示、CI/CD 环境
- **Redis 模式**：长期使用、团队协作、生产部署
