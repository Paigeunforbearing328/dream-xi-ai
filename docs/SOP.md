## 比赛流程（Feature Lifecycle）

每个功能（Feature）遵循以下生命周期：

```
开球(Kickoff) → 热身(Warmup) → 进攻(In-Play) → 审查(Review) → 进球(Goal!)
```

### 1. 开球（Kickoff）

- 主教练提出需求或愿景
- 队长 (#10 Leo) 接球，进行需求分析和任务分解
- 输出：任务卡片（Issue/Card），包含目标、验收标准、预估复杂度

### 2. 热身（Warmup）

- 队长制定战术计划（设计文档/技术方案）
- 跨位置讨论：中场提供审查视角，前锋提供创意方案
- 输出：批准的技术方案，写入 `docs/features/` 目录

### 3. 进攻（In-Play）

- 分配执行：按位置特长分配子任务
- 执行时遵循 TDD（测试驱动开发）：先写测试，再写实现
- 每个球员在自己的战术板（线程）中工作
- 定期同步进度

### 4. 审查（Review）

- **跨位置审查**必须执行：
  - 中场 (#8) 审查队长 (#10) 的代码
  - 队长 (#10) 审查前锋 (#9) 的代码
  - 后卫 (#4) 审查中场 (#8) 的代码
- 审查发现分级：P1（红牌/阻断）、P2（黄牌/应修复）、P3（角球/建议）
- P1 必须修复后才能进入下一阶段

### 5. 进球（Goal!）

- 通过所有门禁检查
- 合并到主分支
- 更新 CHANGELOG.md
- 主教练确认验收

## 传球规则（A2A 通信）

### 基本传球

```
@leo 请分析一下这个需求的技术可行性
@andre 请审查 Leo 刚提交的 PR
@flash 请出一个 UI 设计方案
@wall 请确认基础设施就绪
```

### 结构化交接（Structured Handoff）

当一个球员将任务传给另一个球员时，必须包含：

```markdown
## 交接摘要
- **当前状态**：已完成 X，待完成 Y
- **上下文**：关键决策 A、约束 B
- **待办**：具体需要接手球员做的事
- **参考**：相关文件链接
```

### 冲突解决

当球员之间意见不一致时：

1. 先各自陈述理由（带证据）
2. 如果仍无法达成一致，升级给主教练
3. 主教练做最终决策
4. 决策记录写入共享记忆

## 门禁检查（Quality Gates）

每次合并前必须通过以下检查：

### 赛前检查（Pre-Merge）

```bash
pnpm gate
```

包含：

| 检查项 | 说明 |
|--------|------|
| 代码构建 | `pnpm build` 无错误 |
| 单元测试 | `pnpm test` 全部通过 |
| Lint 检查 | Biome lint 无警告 |
| 类型检查 | TypeScript 严格模式无错误 |
| 跨位置审查 | 至少一名其他位置球员审查 |

### 门将防线（Goalkeeper Checks）

自动化最后一道防线：

- 依赖安全扫描
- 敏感信息检测（API Key、密码等）
- 破坏性变更检测
- 球队铁律合规检查

## 赛后复盘（Post-Match Review）

每个重要功能完成后，进行赛后复盘：

### 复盘模板

```markdown
## 赛后复盘 — [功能名称]

### 比分（结果）
- 完成度：X%
- 耗时：预估 Y 天 / 实际 Z 天

### 精彩回放（做得好的）
- ...

### 失误分析（需改进的）
- ...

### 战术调整（下次改进）
- ...

### 经验沉淀
- 写入 `docs/public-lessons.md`
```

## 紧急响应（Injury Time Protocol）

生产环境紧急问题的处理流程：

1. **报警** — 任何球员发现问题立即在专用线程报告
2. **分诊** — 队长评估严重程度：P0（比赛中断）/ P1（功能受损）/ P2（体验降级）
3. **抢修** — 指定最合适的球员处理，其他球员提供支援
4. **验证** — 修复后必须通过门禁检查
5. **复盘** — 事后写入经验沉淀，避免同类问题再发生
